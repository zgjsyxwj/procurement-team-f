# 系统基础模块要求

## 一、文档目的

本文档用于汇总采购管理平台除核心采购业务外必须具备的系统基础模块要求，作为后续详细设计、研发拆分、数据建模和验收测试的基础输入。

基础模块覆盖：邮件日志、文件服务、操作审计、统一编号、登录日志、SAML/SSO、WorkLife BPM 审批/工作流集成、用户/组织/成本中心同步、功能权限、角色、菜单和页面功能权限管理。

## 二、基础模块清单

| 模块 | 要求状态 | 说明 |
|------|----------|------|
| 邮件发送与邮件日志 | 明确要求 | 系统邮件发送必须记录日志，支持失败追踪和重发 |
| 文件服务 | 明确要求 | 所有业务附件统一通过文件服务上传、存储、下载和权限控制 |
| 操作审计 | 明确要求 | 关键业务操作、数据变更、审批动作、下载行为必须可追溯 |
| 编号前缀与 CodeID 管理 | 明确要求 | 业务实体编号按统一前缀和规则生成，确保唯一 |
| 登录日志 | 需补充明确 | 现有文档已有登录失败次数、最后登录时间和 LOGIN 审计动作，应补充独立登录日志要求 |
| SAML/SSO 登录 | 需补充明确 | 现有文档要求接入 WorkLife/Worktile SSO，协议需明确为 SAML 2.0 或 OAuth2/OIDC |
| WorkLife BPM 审批/工作流集成 | 明确要求 | PR、PO 审批对接 WorkLife BPM，合同审批保留外部审批集成能力 |
| 用户、组织与成本中心同步 | 需补充明确 | 用户、部门、岗位、上下级、成本中心通过其他库同步，数据源由 Lance 提供 |
| 功能权限/角色/菜单/页面动作权限 | 需补充明确 | 需形成统一权限模型，覆盖用户、角色、菜单、页面和动作 |
| 数据权限 | 待定 | 是否按本人、本部门、成本中心、采购员分配关系或其他规则控制，待业务确认 |

## 三、邮件发送与邮件日志模块

### 3.1 功能要求

1. 系统 SHALL 提供统一邮件发送服务，所有业务模块不得直接调用 SMTP。
2. 系统 SHALL 支持企业邮箱 SMTP 配置。
3. 系统 SHALL 支持邮件模板渲染，模板至少包含邮件类型、标题、正文、变量参数。
4. 系统 SHALL 对所有邮件发送操作创建邮件日志，不论发送成功或失败。
5. 系统 SHALL 支持查询邮件发送记录。
6. 系统 SHALL 支持对失败邮件进行手动重发。
7. 系统 SHALL 保留邮件内容快照，用于审计追溯。

### 3.2 日志字段要求

邮件日志至少包含：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| email_type / notification_type | 邮件/通知类型 |
| biz_type | 关联业务类型，如 SUPPLIER、RFQ、PR、PO、CONTRACT |
| biz_id | 关联业务 ID |
| recipient_email | 收件人邮箱 |
| recipient_name | 收件人姓名 |
| subject | 邮件标题 |
| content_snapshot | 邮件内容快照 |
| status | SUCCESS、FAILED、PENDING |
| error_message | 失败原因 |
| retry_count | 重试次数 |
| sent_at | 发送时间 |
| created_at | 创建时间 |

## 四、文件服务模块

### 4.1 功能要求

1. 系统 SHALL 提供统一文件服务，覆盖上传、下载、预览、删除/作废、权限校验和元数据管理。
2. 系统 SHALL 使用阿里云 OSS 存储业务文件。
3. 系统 SHALL 不向前端暴露 OSS 永久公开地址。
4. 系统 SHALL 通过服务端生成带权限控制的临时签名 URL。
5. 系统 SHALL 对文件上传执行格式、大小和风险类型校验。
6. 系统 SHALL 禁止上传可执行文件、脚本文件、安装包和含宏高风险文件。
7. 系统 SHALL 记录文件上传、下载和访问审计。
8. 系统 SHALL 支持业务附件与业务对象绑定，如供应商证件、报价附件、PR/PO 附件、TNC 附件、合同归档文件。

### 4.2 文件元数据字段要求

文件元数据至少包含：

| 字段 | 说明 |
|------|------|
| file_id | 文件唯一 ID |
| original_name | 原始文件名 |
| bucket | OSS Bucket |
| object_key | OSS Object Key |
| mime_type | 文件 MIME 类型 |
| file_ext | 文件扩展名 |
| file_size | 文件大小 |
| file_md5 / file_hash | 文件哈希 |
| biz_type | 业务对象类型 |
| biz_id | 业务对象 ID |
| uploaded_by | 上传人 |
| uploaded_at | 上传时间 |
| status | ACTIVE、DELETED、ARCHIVED |
| tenant_id | 租户 ID |

## 五、操作审计模块

### 5.1 功能要求

1. 系统 SHALL 对关键业务操作记录操作审计。
2. 系统 SHALL 对数据创建、修改、删除、审批、驳回、导出、下载、登录等动作记录审计。
3. 系统 SHALL 对关键字段变更记录变更前值和变更后值。
4. 系统 SHALL 支持按用户、模块、业务对象、操作类型、时间范围查询审计记录。
5. 系统 SHALL 防止审计日志被普通业务用户修改或删除。
6. 系统 SHALL 对敏感字段执行日志脱敏或限制明文记录。

### 5.2 审计字段要求

操作审计至少包含：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| user_id | 操作人 ID |
| user_name | 操作人名称 |
| user_role | 操作人角色 |
| action | 操作类型 |
| module | 功能模块 |
| biz_type | 业务对象类型 |
| biz_id | 业务对象 ID |
| old_value | 变更前数据，JSON |
| new_value | 变更后数据，JSON |
| description | 操作描述 |
| ip_address | IP 地址 |
| user_agent | 浏览器 UA |
| created_at | 操作时间 |
| tenant_id | 租户 ID |

## 六、编号前缀与 CodeID 管理模块

### 6.1 功能要求

1. 系统 SHALL 提供统一编号生成服务。
2. 系统 SHALL 支持按业务实体配置编号前缀、日期格式、流水位数和重置周期。
3. 系统 SHALL 保证编号全局唯一或在指定业务范围内唯一。
4. 系统 SHALL 支持按自然月重置流水号。
5. 系统 SHALL 支持多租户隔离，租户可作为隐含分段键。
6. 系统 SHALL 防止并发生成重复编号。
7. 系统 SHALL 支持草稿预占编号和正式生效编号的业务场景。

### 6.2 默认编号规则

| 业务对象 | 前缀 | 示例 | 规则 |
|----------|------|------|------|
| 供应商 | VD | VD 0001 | VD + 4 位自增序号 |
| 采购申请单 | PR | PR-202605-00001 | PR-YYYYMM-5 位自增 |
| 询价单 | RFQ | RFQ-202605-00001 | RFQ-YYYYMM-5 位自增 |
| 报价单 | QT | QT-202605-00001-A01 | QT-YYYYMM-5 位自增 + 供应商后缀 |
| 合同申请 | CA | CA-202605-00001 | CA-YYYYMM-5 位自增 |
| 正式合同 | CT | CT-202605-00001 | 电子签回传或线下录入 |
| 采购订单 | PO | PO-202605-00001 | PO-YYYYMM-5 位自增 |
| 审批单 | AP | AP-202605-00001 | AP-YYYYMM-5 位自增 |
| 变更记录 | CL | CL-202605-00001 | CL-YYYYMM-5 位自增 |
| 证件 | CE | CE-202605-00001 | CE-YYYYMM-5 位自增 |
| 联系人 | CO | CO-202605-00001 | CO-YYYYMM-5 位自增 |
| 轮次 | RD | RD-1 | RFQ 内按轮次递增 |

## 七、登录日志模块

### 7.1 功能要求

1. 系统 SHALL 记录所有登录尝试，包括成功和失败。
2. 系统 SHALL 记录供应商账号、内部账号和 SSO 登录的登录日志。
3. 系统 SHALL 记录登录失败原因，如账号不存在、密码错误、账号锁定、账号停用、SSO 校验失败。
4. 系统 SHALL 支持连续登录失败次数统计。
5. 系统 SHALL 支持达到最大失败次数后锁定账号。
6. 系统 SHALL 支持登录日志查询，用于安全审计。
7. 系统 SHALL 记录登出、会话超时、Token 失效等会话事件。

### 7.2 登录日志字段要求

登录日志至少包含：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| user_id | 用户 ID，可为空 |
| login_account | 登录账号 |
| login_type | PASSWORD、SSO_SAML、SSO_OIDC |
| result | SUCCESS、FAILED |
| fail_reason | 失败原因 |
| ip_address | IP 地址 |
| user_agent | 浏览器 UA |
| session_id | 会话 ID |
| login_at | 登录时间 |
| logout_at | 登出时间，可为空 |
| tenant_id | 租户 ID |

## 八、SAML/SSO 登录模块

### 8.1 功能要求

1. 系统 SHALL 支持内部用户通过 WorkLife/Worktile SSO 登录。
2. 系统 SHALL 明确 SSO 协议，支持 SAML 2.0 或 OAuth2/OIDC，最终协议以 WorkLife/Worktile 能力为准。
3. 如采用 SAML 2.0，系统 SHALL 支持 SP 发起登录、IdP 回调、断言校验、用户映射和登出处理。
4. 如采用 OAuth2/OIDC，系统 SHALL 支持授权码登录、Token 校验、用户信息拉取和刷新机制。
5. 系统 SHALL 支持将 SSO 用户与内部用户表映射。
6. 系统 SHALL 对 SSO 登录失败记录登录日志。
7. 系统 SHALL 支持 SSO 用户首次登录时自动创建或绑定本地用户，具体规则由用户组织同步模块提供。

### 8.2 用户映射字段要求

内部用户表需支持：

| 字段 | 说明 |
|------|------|
| sso_source | SSO 来源，如 WORKLIFE、WORKTILE |
| sso_user_id | SSO 用户唯一 ID |
| employee_no | 员工工号 |
| email | 邮箱 |
| phone | 手机号 |
| org_id | 所属组织 ID |
| status | 用户状态 |

## 九、WorkLife BPM 审批/工作流集成模块

### 9.1 功能要求

1. 系统 SHALL 对接 WorkLife BPM 审批系统。
2. PR 审批 SHALL 由采购系统发起到 WorkLife BPM，并接收审批结果回调。
3. PO 审批 SHALL 由采购系统发起到 WorkLife BPM，并接收审批结果回调。
4. 合同审批 SHALL 保留外部审批集成能力，可对接 WorkLife BPM 或现有审批平台。
5. 系统 SHALL 对审批请求和回调执行 API Key 或签名认证。
6. 系统 SHALL 支持审批回调幂等处理。
7. 系统 SHALL 记录审批请求日志、审批回调日志和审批状态变更日志。
8. 系统 SHALL 在审批通过、驳回、异常时更新业务单据状态。
9. 系统 SHALL 在审批驳回时保存驳回原因。
10. 系统 SHALL 支持审批异常后的人工重试或管理员重放处理。

### 9.2 集成日志字段要求

外部请求日志至少包含：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| system_code | 外部系统编码，如 WORKLIFE_BPM |
| request_type | 请求类型 |
| biz_type | 业务类型 |
| biz_id | 业务 ID |
| request_payload | 请求报文 |
| response_payload | 响应报文 |
| status | SUCCESS、FAILED |
| error_message | 错误信息 |
| duration_ms | 耗时 |
| idempotency_key | 幂等键 |
| created_at | 创建时间 |

外部回调日志至少包含：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| system_code | 外部系统编码 |
| callback_type | 回调类型 |
| raw_payload | 原始回调报文 |
| signature | 签名信息 |
| process_status | 处理状态 |
| error_message | 异常原因 |
| biz_type | 业务类型 |
| biz_id | 业务 ID |
| received_at | 接收时间 |
| processed_at | 处理时间 |

## 十、用户、组织与成本中心同步模块

### 10.1 功能要求

1. 系统 SHALL 支持从外部用户组织库同步用户、部门、岗位、组织关系和成本中心。
2. 外部用户组织库、成本中心库、字段口径、同步方式和访问凭据由 Lance 提供。
3. 系统 SHALL 支持按员工工号、手机号、邮箱、用户名或 SSO 用户 ID 映射内部用户。
4. 系统 SHALL 支持组织关系同步，包括部门、上级部门、用户所属部门、用户直属上级。
5. 系统 SHALL 支持成本中心同步，包括成本中心编码、名称、状态、负责人和代理人。
6. 系统 SHALL 支持用户与成本中心关系同步，用户表中的 cost_center_id SHALL 指向成本中心表主键。
7. 系统 SHALL 支持用户状态同步，包括在职、离职、停用。
8. 系统 SHALL 支持采购员、采购经理、业务人员等系统角色与组织用户绑定。
9. 系统 SHALL 支持同步失败记录和重试。
10. 系统 SHALL 保留业务单据中的用户、组织和成本中心名称快照，避免主数据变更影响历史记录。

### 10.2 同步对象

| 对象 | 说明 |
|------|------|
| User | 内部用户、员工、供应商账号 |
| Organization | 部门、组织单元 |
| UserOrganization | 用户所属部门关系 |
| UserManager | 用户上下级关系 |
| Position | 岗位，可选 |
| CostCenter | 成本中心 |
| UserCostCenter | 用户与成本中心关系 |
| RoleBinding | 用户与系统角色绑定 |

### 10.3 用户同步字段要求

用户同步目标表可参考 `users`，至少包含：

| 字段 | 说明 |
|------|------|
| id | 用户主键，来自外部库或同步映射生成 |
| username | 用户名 |
| email | 邮箱 |
| name_en | 英文名 |
| name_zh | 中文名 |
| workid | 工号，用户同步关键字段之一 |
| work_email | 工作邮箱 |
| full_display | 展示名称 |
| cost_center_id | 成本中心 ID，关联 cost_centers.id |
| lm_user_id | 直线经理用户 ID，关联 users.id |

### 10.4 成本中心同步字段要求

成本中心同步目标表可参考 `cost_centers`，至少包含：

| 字段 | 说明 |
|------|------|
| id | 成本中心主键 |
| code | 成本中心编码，同步唯一键之一 |
| name | 成本中心名称 |
| status | 状态，0 表示不启用，1 表示启用 |
| director_user_id | 负责人用户 ID，关联 users.id |
| agent_user_nos | 代理人工号，多个用逗号分隔，对应 users.workid |

### 10.5 关联与校验规则

1. 用户同步时，系统 SHALL 按外部成本中心编码或 ID 解析并写入 users.cost_center_id。
2. 成本中心同步时，系统 SHALL 按负责人唯一标识解析并写入 cost_centers.director_user_id。
3. 成本中心代理人 SHALL 按工号维护，多个代理人工号使用逗号分隔；系统 SHOULD 在同步后校验工号是否存在。
4. 用户直线经理 SHALL 按外部经理标识解析并写入 users.lm_user_id。
5. 若成本中心不存在，用户同步 SHALL 记录异常并按配置决定跳过该用户、创建占位成本中心或置空 cost_center_id。
6. 若负责人或直线经理不存在，系统 SHALL 记录同步异常，不得阻断其他用户和成本中心同步。
7. 已停用成本中心不得作为新 PR/PO 的可选成本中心，但历史单据快照仍可展示。

### 10.6 同步策略

1. 系统 SHALL 支持定时全量同步。
2. 系统 SHOULD 支持增量同步或事件回调同步。
3. 系统 SHALL 对外部唯一 ID 建立唯一约束。
4. 系统 SHALL 对已离职或停用用户禁用登录和新业务操作。
5. 系统 SHALL 不物理删除已参与业务流程的历史用户。
6. 系统 SHALL 不物理删除已被历史 PR/PO 引用的成本中心。
7. 系统 SHALL 记录每次同步批次、同步来源、开始时间、结束时间、成功数量、失败数量和失败明细。

## 十一、用户、角色、菜单、页面和动作权限管理模块

### 11.1 权限模型

系统 SHALL 采用基于用户、角色、菜单、页面和动作的权限模型。

数据权限暂不纳入本权限模型，具体控制口径待定。

基础对象包括：

| 对象 | 说明 |
|------|------|
| User | 用户 |
| Role | 角色 |
| Menu | 菜单 |
| Page | 页面 |
| Action | 页面动作，如新增、编辑、删除、提交、审批、导出、下载 |
| UserRole | 用户角色关系 |
| RoleMenu | 角色菜单关系 |
| RolePage | 角色页面关系 |
| RoleAction | 角色动作关系 |

### 11.2 角色要求

系统默认角色包括：

| 角色编码 | 角色名称 |
|----------|----------|
| ADMIN | 管理员/采购经理 |
| BUYER | 采购员 |
| BUSINESS_USER | 业务人员 |
| SUPPLIER | 供应商 |

### 11.3 菜单权限要求

1. 系统 SHALL 支持按角色配置菜单可见性。
2. 系统 SHALL 支持多级菜单。
3. 系统 SHALL 支持菜单排序、启用、停用。
4. 系统 SHALL 支持菜单与页面路由绑定。
5. 用户登录后 SHALL 只返回当前用户有权限访问的菜单。

### 11.4 页面与动作权限要求

1. 系统 SHALL 支持页面级权限控制。
2. 系统 SHALL 支持动作级权限控制，如新增、编辑、删除、提交、审批、导出、下载、重发邮件。
3. 系统 SHALL 支持按角色配置页面可访问范围。
4. 系统 SHALL 支持按角色配置页面内可执行动作。
5. 用户进入页面后 SHALL 只展示当前用户可执行的动作入口。
6. 系统 SHALL 对无权限访问返回 403。
7. 系统 SHALL 对越权访问记录安全审计。

### 11.5 数据权限待定

数据权限暂不在本章节展开，待业务确认后再补充。待确认范围包括：

1. 是否按本人、本部门、成本中心、采购员分配关系、供应商归属关系控制数据。
2. Admin、Buyer、Business_User、Supplier 的数据可见范围。
3. 报表、列表、详情、导出是否使用相同的数据权限规则。
4. 数据权限是否需要支持后台配置。

### 11.6 动作编码建议

动作编码建议格式：

```text
页面:动作
```

示例：

| 动作编码 | 说明 |
|--------|------|
| supplier_company:view | 查看供应商 |
| supplier_company:create | 创建供应商 |
| supplier_company:update | 编辑供应商 |
| supplier_certificate:approve | 审核证件 |
| rfq:create | 创建询价单 |
| quote:return | 退回报价 |

## 十二、验收要求

1. 任一系统邮件发送后，均可在邮件日志中查询到对应记录。
2. 任一文件上传后，均生成文件元数据和可追踪的 OSS objectKey。
3. 任一文件下载均经过权限校验，并记录下载审计。
4. 任一关键业务操作均可在操作审计中追溯操作人、操作时间和业务对象。
5. 并发创建业务单据时，不得生成重复业务编号。
6. 用户登录成功、失败、锁定、登出和会话超时均可追踪。
7. 内部用户可通过 WorkLife/Worktile SSO 登录，并完成本地用户映射。
8. PR 和 PO 可发起 WorkLife BPM 审批，并能正确处理审批通过、驳回和重复回调。
9. 外部用户组织库和成本中心库同步后，系统用户、组织、上下级、成本中心和用户成本中心关系保持一致。
10. 不同角色登录后只能看到有权限的菜单、页面和动作。
11. 无权限用户访问菜单、页面或执行动作时，系统返回无权限提示并记录越权审计。
